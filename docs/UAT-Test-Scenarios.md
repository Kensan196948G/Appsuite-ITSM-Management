# UAT（ユーザー受け入れテスト）シナリオ

**作成日**: 2026-01-29
**バージョン**: 1.0.0
**対象システム**: AppSuite ITSM管理システム

---

## 概要

本ドキュメントはエンドユーザーによる受け入れテスト（UAT）のシナリオを定義します。
各シナリオは実際の業務フローに基づいており、システムが要件を満たしていることを確認します。

## テスト環境

| 項目 | 設定 |
|------|------|
| URL | http://localhost:3000（または本番URL） |
| ブラウザ | Chrome 90+, Firefox 88+, Safari 14+, Edge 90+ |
| テストアカウント | admin / admin123（管理者）、demo / demo123（一般ユーザー） |

## 詳細シナリオ

### UAT-001: 管理者ログインと初期確認

**目的**: システムへの正常なログインとダッシュボード表示を確認

**前提条件**: なし

**手順**:

| ステップ | 操作 | 期待結果 |
|---------|------|---------|
| 1 | ブラウザでシステムURLを開く | ログイン画面が表示される |
| 2 | ユーザー名「admin」を入力 | 入力が反映される |
| 3 | パスワード「admin123」を入力 | パスワードがマスク表示される |
| 4 | 「ログイン」ボタンをクリック | ダッシュボードに遷移 |
| 5 | ダッシュボードの統計カードを確認 | ユーザー数、アプリ数等が表示 |
| 6 | KPI指標を確認 | 解決時間、SLA達成率等が表示 |
| 7 | 最近の操作ログを確認 | ログインが記録されている |

**合格基準**:
- [ ] ログイン成功トースト表示
- [ ] 全統計カードにデータ表示
- [ ] KPIウィジェット表示
- [ ] 操作ログに自身のログイン記録

---

### UAT-002: ユーザー管理CRUD操作

**目的**: ユーザーの作成・読取・更新・削除を確認

**前提条件**: 管理者としてログイン済み

**手順**:

| ステップ | 操作 | 期待結果 |
|---------|------|---------|
| 1 | サイドバー「ユーザー管理」をクリック | ユーザー一覧が表示 |
| 2 | 「新規ユーザー」ボタンをクリック | ユーザー作成モーダル表示 |
| 3 | 以下を入力:<br>- ユーザー名: テスト太郎<br>- メール: test@example.com<br>- 部署: テスト部<br>- 権限: ユーザー | 各フィールドに入力反映 |
| 4 | 「保存」ボタンをクリック | モーダルが閉じ、一覧に追加 |
| 5 | 作成したユーザーの編集ボタンをクリック | 編集モーダル表示 |
| 6 | 部署を「営業部」に変更して保存 | 一覧で部署が更新 |
| 7 | 検索ボックスに「テスト」と入力 | フィルタリングで該当ユーザー表示 |
| 8 | 作成したユーザーの削除ボタンをクリック | 確認ダイアログ表示 |
| 9 | 削除を確認 | ユーザーが一覧から削除 |

**合格基準**:
- [ ] ユーザー作成成功
- [ ] ユーザー編集成功
- [ ] 検索・フィルタ動作
- [ ] ユーザー削除成功
- [ ] 各操作が監査ログに記録

---

### UAT-003: インシデント起票から解決まで

**目的**: インシデント管理の全ライフサイクルを確認

**前提条件**: 管理者としてログイン済み

**手順**:

| ステップ | 操作 | 期待結果 |
|---------|------|---------|
| 1 | 「インシデント管理」をクリック | インシデント一覧表示 |
| 2 | 「新規インシデント」をクリック | 作成モーダル表示 |
| 3 | 以下を入力:<br>- タイトル: UATテストインシデント<br>- 対象アプリ: 勤怠管理<br>- 優先度: 高<br>- 説明: テスト用のインシデント | 入力が反映される |
| 4 | 「保存」ボタンをクリック | 一覧に「オープン」で追加 |
| 5 | 作成したインシデントの編集ボタンをクリック | 編集モーダル表示 |
| 6 | ステータスを「対応中」に変更、担当者を設定 | 変更が反映される |
| 7 | 「保存」をクリック | ステータスが「対応中」に更新 |
| 8 | 再度編集し、ステータスを「解決済」に変更 | 変更が反映される |
| 9 | 「保存」をクリック | ステータスが「解決済」に更新 |
| 10 | 再度編集し、ステータスを「クローズ」に変更 | 変更が反映される |

**ワークフロー検証**:

```
オープン → 対応中 → 解決済 → クローズ
   ✓         ✓         ✓         ✓
```

**合格基準**:
- [ ] インシデント作成成功
- [ ] ステータス遷移が正常（ワークフロールール遵守）
- [ ] 担当者割り当て動作
- [ ] 通知が発生（高優先度の場合）
- [ ] 監査ログに全操作記録

### UAT-004: 変更要求の承認フロー

**目的**: 変更管理の承認フローを確認

**前提条件**: 管理者としてログイン済み

**手順**:

| ステップ | 操作 | 期待結果 |
|---------|------|---------|
| 1 | 「変更管理」をクリック | 変更要求一覧表示 |
| 2 | 「新規変更要求」をクリック | 作成モーダル表示 |
| 3 | 以下を入力:<br>- タイトル: UAT機能追加要求<br>- 対象アプリ: 顧客管理<br>- 種別: 機能追加<br>- 実施予定日: 来週 | 入力が反映される |
| 4 | 「保存」ボタンをクリック | 一覧に「下書き」で追加 |
| 5 | 編集でステータスを「申請中」に変更 | 承認待ち状態に |
| 6 | 編集でステータスを「承認」に変更 | 承認済み状態に |
| 7 | 編集でステータスを「実施中」に変更 | 実施中状態に |
| 8 | 編集でステータスを「完了」に変更 | 完了状態に |

**ワークフロー検証**:

```
下書き → 申請中 → 承認 → 実施中 → 完了
   ✓        ✓       ✓       ✓       ✓
```

**合格基準**:
- [ ] 変更要求作成成功
- [ ] 承認フロー遷移が正常
- [ ] 不正な遷移（例: 下書き→完了）がブロックされる
- [ ] 監査ログに全操作記録

### UAT-005: 監査ログ検索とエクスポート

**目的**: 監査ログの検索・フィルタ・エクスポート機能を確認

**前提条件**: 管理者としてログイン済み、いくつかの操作を実行済み

**手順**:

| ステップ | 操作 | 期待結果 |
|---------|------|---------|
| 1 | 「監査ログ」をクリック | ログ一覧表示 |
| 2 | 操作タイプフィルタで「ログイン」を選択 | ログイン操作のみ表示 |
| 3 | 対象フィルタで「インシデント」を選択 | インシデント関連操作のみ表示 |
| 4 | フィルタをクリア | 全ログ表示に戻る |
| 5 | 「エクスポート」ボタンをクリック | エクスポートオプション表示 |
| 6 | CSV形式でエクスポート | CSVファイルがダウンロード |
| 7 | JSON形式でエクスポート | JSONファイルがダウンロード |

**合格基準**:
- [ ] フィルタ機能が正常動作
- [ ] CSVエクスポート成功
- [ ] JSONエクスポート成功
- [ ] エクスポートファイルの内容が正確

### UAT-006: バックアップと復元

**目的**: データのバックアップと選択的復元を確認

**前提条件**: 管理者としてログイン済み

**手順**:

| ステップ | 操作 | 期待結果 |
|---------|------|---------|
| 1 | 「システム設定」→「バックアップ」タブ | バックアップ設定画面表示 |
| 2 | 現在のデータサイズを確認 | サイズが表示される |
| 3 | 「バックアップ作成」をクリック | JSONファイルがダウンロード |
| 4 | ダウンロードしたファイルの内容を確認 | ユーザー、アプリ等のデータ含む |
| 5 | テストユーザーを1件削除 | ユーザーが削除される |
| 6 | 「ファイルを選択」でバックアップファイルを選択 | リストアオプション表示 |
| 7 | 「ユーザー」のみ選択してリストア | ユーザーデータのみ復元 |
| 8 | ユーザー一覧を確認 | 削除したユーザーが復元 |

**合格基準**:
- [ ] バックアップファイル生成成功
- [ ] ファイルにバージョン・タイムスタンプ含む
- [ ] 選択的リストアが動作
- [ ] リストア後のデータ整合性確認

### UAT-007: 通知機能の動作確認

**目的**: ブラウザ通知と通知履歴を確認

**前提条件**: 管理者としてログイン済み、ブラウザ通知を許可

**手順**:

| ステップ | 操作 | 期待結果 |
|---------|------|---------|
| 1 | ヘッダーの通知ベルを確認 | 未読バッジ表示 |
| 2 | 通知ベルをクリック | 通知パネル表示 |
| 3 | 通知履歴を確認 | 過去の通知一覧表示 |
| 4 | 高優先度インシデントを作成 | ブラウザ通知が表示 |
| 5 | 通知パネルを再確認 | 新しい通知が追加 |
| 6 | 「既読にする」をクリック | 未読バッジ減少 |

**合格基準**:
- [ ] 通知ベルのバッジ表示
- [ ] 通知パネル表示
- [ ] 新規インシデント通知発生
- [ ] 通知履歴の表示・管理

### UAT-008: SLA監視とエスカレーション

**目的**: SLA違反検出と自動エスカレーションを確認

**前提条件**: 管理者としてログイン済み

**手順**:

| ステップ | 操作 | 期待結果 |
|---------|------|---------|

| 3 | 通知を確認 | SLA違反通知あり |
| 4 | 「インシデント管理」を確認 | エスカレート済みマーク |
| 5 | 監査ログを確認 | エスカレーション記録あり |

**SLA基準**:

| 優先度 | 応答時間 | 解決時間 |
|--------|---------|---------|
| 高 | 1時間 | 4時間 |
| 中 | 4時間 | 24時間 |
| 低 | 24時間 | 72時間 |

**合格基準**:
- [ ] SLA超過検出
- [ ] 警告通知発生
- [ ] 自動エスカレーション実行
- [ ] 監査ログへの記録
