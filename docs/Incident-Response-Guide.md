## 1. 概要

### 1.1 目的

本ガイドは、AppSuite管理運用システムで発生したインシデント（障害・不具合・問い合わせ）に対して、
迅速かつ適切に対応するための手順を定めます。

### 1.2 適用範囲

本ガイドは、以下のインシデントに適用されます:

- システム障害（ログイン不可、画面表示不可等）
- データ問題（データ損失、不整合等）
- セキュリティインシデント（不正アクセス、情報漏洩等）
- 性能問題（画面遅延、タイムアウト等）
- ユーザーからの問い合わせ・エラー報告

### 1.3 インシデント管理の原則

**迅速性**: 可能な限り早く対応を開始する
**透明性**: 関係者に状況を適切に報告する
**記録**: 全ての対応を記録する
**再発防止**: 根本原因を特定し、恒久対策を実施する

## 2. インシデントレベル定義

### 2.1 レベル分類基準

| レベル | 定義 | 影響範囲 | 業務影響 | 例 |
|--------|------|---------|---------|-----|
| **L1（重大）** | システム全体が停止 | 全ユーザー | 業務完全停止 | ・システムアクセス不可<br>・全データ消失<br>・不正アクセスによるデータ漏洩 |
| **L2（高）** | 主要機能が停止 | 多数のユーザー | 業務に重大な支障 | ・ログイン不可<br>・インシデント登録不可<br>・バックアップ・リストア不可 |
| **L3（中）** | 一部機能に問題 | 一部のユーザー | 業務継続可能だが支障あり | ・検索機能が動かない<br>・エクスポートエラー<br>・画面表示崩れ |
| **L4（低）** | 軽微な問題 | 限定的 | 業務への影響は小さい | ・誤字脱字<br>・表示の微調整<br>・改善要望 |

### 2.2 対応時間目標（SLA）

| レベル | 検知・報告 | 初動対応開始 | 復旧目標時間 | 解決目標時間 | 報告先 |
|--------|-----------|-------------|-------------|-------------|--------|
| **L1** | 即座 | 5分以内 | 1時間以内 | 4時間以内 | 経営層・全関係者 |
| **L2** | 15分以内 | 15分以内 | 4時間以内 | 8時間以内 | 管理者・関係者 |
| **L3** | 1時間以内 | 2時間以内 | 24時間以内 | 72時間以内 | 管理者 |
| **L4** | 1営業日以内 | 3営業日以内 | - | 1週間以内 | 担当者 |

### 2.3 優先度の判断フローチャート

```
インシデント発生
  ↓
┌─────────────────────────┐
│ 全ユーザーに影響？        │→ YES → L1（重大）
└─────────────────────────┘
  ↓ NO
┌─────────────────────────┐
│ 主要機能が停止？          │→ YES → L2（高）
│ または多数のユーザーに影響？│
└─────────────────────────┘
  ↓ NO
┌─────────────────────────┐
│ 回避策がない？            │→ YES → L3（中）
│ または業務に支障？         │
└─────────────────────────┘
  ↓ NO
└──→ L4（低）
```

## 3. レベル別対応手順

### 3.1 L1（重大）インシデント対応

#### 3.1.1 初動対応（5分以内）

**Step 1: 検知・確認（1分）**
```
□ インシデントの事実確認（再現確認）
□ 影響範囲の概算（全ユーザー/一部ユーザー）
□ 発生時刻の記録
```

**Step 2: 緊急連絡（2分）**
```
□ システム管理者に電話連絡
□ PM・経営層にメール+電話連絡
□ 関係者にチャット・メールで一斉通知
```

**連絡内容**:
```
【重大インシデント発生】
・インシデント内容: ___________
・影響範囲: 全ユーザー
・発生時刻: YYYY/MM/DD HH:MM
・現在の状況: ___________
・対応者: ___________
```

**Step 3: インシデント記録（2分）**
```
□ システムにインシデント番号を発番（INC-YYYY-XXXX）
□ 優先度「L1」を設定
□ 詳細情報を記録
□ ステータスを「対応中」に設定
```

#### 3.1.2 復旧作業（1時間以内）

**Step 4: 応急処置（15分以内）**
```
□ システムが停止している場合 → Webサーバー再起動
□ データ損失の場合 → 最新バックアップからリストア
□ 不正アクセスの場合 → 該当アカウントのロック、アクセス遮断
□ SSL証明書エラーの場合 → 証明書の再インストール
```

**Step 5: 根本原因調査（並行実施）**
```
□ エラーログの確認
  - Webサーバーログ: /var/log/apache2/error.log
  - アプリケーションログ: ブラウザコンソール（F12）
□ システム監査ログの確認
□ 直前の変更内容の確認
□ 外部要因の確認（ネットワーク障害等）
```

**Step 6: 復旧確認（30分以内）**
```
□ システムへのアクセスが可能か
□ 全主要機能が動作するか
  - ログイン
  - ダッシュボード表示
  - インシデント登録
  - バックアップ・リストア
□ データの整合性確認
□ セキュリティ状態の確認
```

**Step 7: ユーザーへの復旧通知（復旧後15分以内）**
```
件名: 【復旧完了】AppSuite管理運用システム

関係者各位

システムの復旧が完了しましたのでお知らせします。

・インシデント番号: INC-YYYY-XXXX
・障害内容: ___________
・発生時刻: YYYY/MM/DD HH:MM
・復旧時刻: YYYY/MM/DD HH:MM
・影響時間: XX時間XX分
・原因: ___________（調査中の場合は「調査中」）
・再発防止策: ___________

ご迷惑をおかけして申し訳ございませんでした。
```

#### 3.1.3 事後対応（4時間以内）

**Step 8: 詳細報告書作成**
```
□ インシデントレポート作成（テンプレート使用）
□ タイムライン作成
□ 根本原因分析（RCA）
□ 再発防止策の立案
□ 関係者への報告
```

**Step 9: 恒久対策の実施計画**
```
□ 変更要求の作成
□ 優先度の設定
□ 実施スケジュールの策定
□ 承認取得
```

### 3.2 L2（高）インシデント対応

#### 3.2.1 初動対応（15分以内）

**Step 1: 検知・確認（5分）**
```
□ インシデントの再現確認
□ 影響範囲の特定（ユーザー数、機能）
□ 回避策の有無を確認
```

**Step 2: 関係者への連絡（5分）**
```
□ システム管理者にメール+電話
□ 影響を受けるユーザーにメールで通知
```

**Step 3: インシデント記録（5分）**
```
□ インシデント番号発番
□ 優先度「L2」設定
□ ステータス「対応中」
```

#### 3.2.2 復旧作業（4時間以内）

**Step 4: 原因調査（1時間以内）**
```
□ ログ分析
□ 再現テスト
□ 直前の変更確認
```

**Step 5: 修正・復旧（2時間以内）**
```
□ 修正コードの作成（必要な場合）
□ テスト環境での動作確認
□ 本番環境への適用
□ 復旧確認
```

**Step 6: ユーザーへの通知**
```
件名: 【復旧完了】〇〇機能の不具合対応

影響を受けた皆様

〇〇機能の不具合が復旧しましたのでお知らせします。

・インシデント番号: INC-YYYY-XXXX
・障害内容: ___________
・復旧時刻: YYYY/MM/DD HH:MM

ご迷惑をおかけして申し訳ございませんでした。
```

### 3.3 L3（中）インシデント対応

#### 3.3.1 初動対応（2時間以内）

**Step 1: インシデント登録（30分）**
```
□ インシデント詳細を記録
□ 優先度「L3」設定
□ 担当者アサイン
```

**Step 2: 回避策の提供（1時間以内）**
```
□ 回避策を検討
□ ユーザーに回避策を案内
```

#### 3.3.2 解決作業（24時間以内）

**Step 3: 原因調査と修正**
```
□ 詳細な原因調査
□ 修正内容の検討
□ テスト実施
□ 本番適用
```

**Step 4: 完了報告**
```
□ ユーザーに解決を報告
□ インシデントをクローズ
```

### 3.4 L4（低）インシデント対応

#### 3.4.1 対応（1週間以内）

**Step 1: インシデント登録**
```
□ インシデント記録
□ 優先度「L4」設定
```

**Step 2: 計画的対応**
```
□ 他の作業と優先度を調整
□ スケジュールに組み込み
□ 対応実施
```

## 4. エスカレーションフロー

### 4.1 エスカレーションパス

```
【L1（重大）】
発見者 → システム管理者（即座）→ PM（即座）→ 経営層（15分以内）
              ↓
         全関係者に一斉通知

【L2（高）】
発見者 → システム管理者（15分以内）→ PM（1時間以内）
              ↓
         影響ユーザーに通知

【L3（中）】
発見者 → 運用担当者（2時間以内）→ システム管理者（必要時）

【L4（低）】
発見者 → 運用担当者（1営業日以内）
```

### 4.2 エスカレーション判断基準

以下の場合、上位レベルにエスカレーション:

```
□ 目標時間内に復旧できない
□ 技術的に解決困難
□ 影響範囲が拡大している
□ セキュリティリスクが判明
□ ユーザーからの強い要望
□ メディア・外部への影響が懸念される
```

### 4.3 エスカレーション時の情報

エスカレーション時に伝える情報:

```
1. インシデント番号
2. レベル
3. 発生時刻
4. 現在の状況
5. 影響範囲
6. これまでの対応内容
7. エスカレーション理由
8. 必要な支援内容
```

## 7. 事後レビュー手順

### 7.1 事後レビューの目的

- 根本原因の特定
- 再発防止策の立案
- 対応プロセスの改善
- ナレッジの共有

### 7.2 レビュー実施基準

| レベル | レビュー実施 | 実施タイミング | 参加者 |
|--------|------------|---------------|--------|
| **L1** | 必須 | 復旧後24時間以内 | 全関係者 |
| **L2** | 必須 | 復旧後3営業日以内 | 関係者 |
| **L3** | 推奨 | 復旧後1週間以内 | 担当者+管理者 |
| **L4** | 任意 | - | 担当者 |

### 7.3 レビュー会議アジェンダ（L1/L2）

**所要時間**: 60分

```
1. インシデント概要説明（5分）
   - 発生日時、影響範囲、復旧時間

2. タイムライン詳細（10分）
   - 発生から復旧までの時系列

3. 根本原因分析（RCA）（15分）
   - Why-Whyを5回繰り返して真因を特定

4. 対応の振り返り（15分）
   - 良かった点
   - 改善すべき点
   - エスカレーションは適切だったか

5. 再発防止策の討議（10分）
   - 技術的対策
   - プロセス改善
   - 教育・トレーニング

6. アクションプラン策定（5分）
   - 担当者、期限、優先度の設定
```

```

インシデント: ___________
```

## 8. よくあるインシデントと対処法

### 8.1 システムアクセス不可

**症状**: ブラウザでシステムURLにアクセスできない

**レベル**: L1（重大）

**初動対応**:
```
□ Webサーバーの稼働状態確認
  sudo systemctl status apache2
  または
  sudo systemctl status nginx

□ エラーログ確認
  sudo tail -n 50 /var/log/apache2/error.log

□ Webサーバー再起動
  sudo systemctl restart apache2
```

**根本原因候補**:
- Webサーバーの停止
- SSL証明書の期限切れ
- 設定ファイルのシンタックスエラー
- ポートの競合

### 8.2 ログインできない

**症状**: ユーザー名・パスワードを入力してもログインできない

**レベル**: L2（高）

**初動対応**:
```
□ 該当ユーザーのアカウント状態確認
  - ステータスが「有効」か
  - パスワードが正しいか

□ ブラウザのJavaScriptが有効か確認

□ 監査ログでログイン試行を確認
  - エラーメッセージの確認

□ 他のユーザーでログインを試す
  - 特定ユーザーの問題か全体の問題かを切り分け
```

**根本原因候補**:
- アカウントが無効化されている
- パスワードポリシー変更の影響
- セッション管理の不具合
- ブラウザのlocalStorage問題

### 8.3 データが保存されない

**症状**: データを入力して保存しても反映されない

**レベル**: L2（高）

**初動対応**:
```
□ ブラウザのlocalStorage容量確認
  - F12 → Application → Local Storage
  - 5MB以上使用している場合は容量超過

□ ブラウザのシークレットモード確認
  - シークレットモードでは保存されない

□ 入力エラーの確認
  - 必須項目が全て入力されているか
  - バリデーションエラーがないか

□ ブラウザコンソールのエラー確認
  - F12 → Console
```

**根本原因候補**:
- localStorage容量超過
- シークレットモード使用
- JavaScriptエラー
- ブラウザのストレージ設定

### 8.4 画面が表示されない・真っ白

**症状**: ページが真っ白で何も表示されない

**レベル**: L2（高）

**初動対応**:
```
□ ブラウザのキャッシュクリア
  - Ctrl + Shift + Delete
  - 「キャッシュされた画像とファイル」を削除

□ 強制再読み込み
  - Ctrl + Shift + R

□ ブラウザコンソールのエラー確認
  - F12 → Console
  - JavaScriptエラーを確認

□ 別のブラウザで確認
```

**根本原因候補**:
- JavaScriptエラー
- ブラウザキャッシュの問題
- ファイルの読み込み失敗
- ブラウザの非対応

### 8.5 API接続エラー

**症状**: ヘッダーに「API未接続」と表示される

**レベル**: L3（中）

**初動対応**:
```
□ システム設定→API接続タブを確認
  - API URL、APIキー、認証方式が正しいか

□ 「接続テスト」を実行
  - エラーメッセージを確認

□ DeskNet's Neoサーバーの稼働確認
  - ブラウザでAPI URLにアクセス

□ ネットワーク接続確認
```

**根本原因候補**:
- APIキーの期限切れ
- DeskNet's Neoサーバーの停止
- ネットワーク問題
- 設定ミス

### 8.6 画面が遅い・重い

**症状**: ページの読み込みや操作が非常に遅い

**レベル**: L3（中）

**初動対応**:
```
□ ブラウザキャッシュクリア

□ データ量確認
  - 監査ログが10,000件以上ある場合は削除

□ サーバーリソース確認
  - CPU、メモリ使用率

□ ネットワーク速度確認

□ ブラウザ拡張機能を無効化
  - 広告ブロッカー等
```

**根本原因候補**:
- データ量過多
- サーバーリソース不足
- ネットワーク遅延
- ブラウザ拡張機能の影響

### 8.7 SSL証明書エラー

**症状**: 「この接続ではプライバシーが保護されません」と表示

**レベル**: L1（重大）

**初動対応**:
```
□ SSL証明書の有効期限確認
  openssl x509 -in /etc/ssl/certs/appsuite-itsm.crt -noout -enddate

□ 証明書のインストール状態確認
  sudo apache2ctl configtest（Apache）
  sudo nginx -t（Nginx）

□ 緊急の場合、HTTP（非暗号化）での一時アクセス案内
  http://appsuite-itsm.example.com/
  （本番環境では非推奨）
```

**根本原因候補**:
- 証明書の期限切れ
- 証明書のインストールミス
- 中間証明書の不足
- サーバー設定の誤り

### 8.8 データ消失

**症状**: 登録したデータが消えた

**レベル**: L1（重大）

**初動対応**:
```
□ 削除操作の有無を監査ログで確認

□ ブラウザのlocalStorageを確認
  - F12 → Application → Local Storage

□ 最新バックアップの確認

□ 別のブラウザやPCでアクセスして確認
  - localStorageはブラウザごとに独立

□ 必要に応じてバックアップからリストア
```

**根本原因候補**:
- 誤削除
- ブラウザのデータクリア
- localStorageの破損
- シークレットモードでの使用
