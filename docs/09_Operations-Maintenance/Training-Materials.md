## 1. トレーニング概要

### 1.1 トレーニングの目的

このトレーニングでは、AppSuite管理運用システムの基本的な使い方を習得し、実務で活用できるスキルを身につけます。

### 1.2 対象者

- **新規ユーザー**: システムを初めて使う方
- **運用担当者**: システムの日常運用を担当する方
- **管理者候補**: 管理者権限を持つ予定の方

### 1.3 前提知識

- 基本的なPC操作スキル
- Webブラウザの使用経験
- ITサービス管理の基礎知識（あれば尚可）

### 1.4 トレーニング後に達成すべきこと

- [ ] システムにログイン・ログアウトできる
- [ ] ダッシュボードの見方を理解している
- [ ] 基本的なCRUD操作（作成・読取・更新・削除）ができる
- [ ] インシデントを登録・対応できる
- [ ] 変更要求を作成・承認できる
- [ ] 監査ログを検索・エクスポートできる
- [ ] バックアップを取得できる

### 1.5 準備物

**受講者**:
- ノートPC（Windows/Mac/Linux）
- Webブラウザ（Chrome推奨）
- インターネット接続
- メモ帳・筆記用具

**講師**:
- プロジェクター/大画面ディスプレイ
- トレーニング用アカウント
- ハンズオン課題資料
- 評価シート

## 3. セッション1: システム概要（15分）

### 3.1 学習目標

- システムの目的と概要を理解する
- 主要機能を把握する
- システム全体の構成を理解する

### 3.2 講義内容

#### 3.2.1 AppSuite管理運用システムとは

**定義**:
DeskNet's Neo AppSuiteで作成された業務アプリケーションを一元管理するWebベースのITSM（ITサービス管理）ツール

**目的**:
- 業務アプリの稼働状況を把握
- インシデント（障害）を迅速に対応
- 変更要求を計画的に管理
- すべての操作を監査ログで追跡

#### 3.2.2 主要機能（6つ）

| 機能 | 説明 | 主な利用者 |
|------|------|-----------|
| **ダッシュボード** | システム全体の統計情報表示 | 全員 |
| **ユーザー管理** | ユーザーアカウントのCRUD操作 | 管理者 |
| **アプリ管理** | AppSuiteアプリの一覧管理 | 管理者・運用担当 |
| **インシデント管理** | 障害・不具合の記録と対応追跡 | 全員 |
| **変更管理** | 機能追加・改善要求の承認フロー | 全員 |
| **監査ログ** | 全操作履歴の記録・検索 | 管理者 |

#### 3.2.3 技術スタック

- **フロントエンド**: HTML5, CSS3, JavaScript
- **データ保存**: ブラウザのlocalStorage
- **外部連携**: DeskNet's Neo REST API（オプション）

#### 3.2.4 アクセス方法

- **本番環境**: `https://appsuite-itsm.example.com/`
- **開発環境**: `http://localhost:8888/`

### 3.3 演習

**確認問題**:

1. このシステムの主な目的は何ですか？（3つ挙げてください）
2. インシデント管理と変更管理の違いを説明してください
3. データはどこに保存されますか？

**解答**:
1. ①業務アプリの稼働状況把握、②インシデントの迅速対応、③変更要求の計画的管理
2. インシデント管理=既に発生した問題の対応、変更管理=新機能追加・改善の申請
3. ブラウザのlocalStorage

#### 4.2.3 ダッシュボードの見方

**統計カード**（6枚）:

| カード | 表示内容 | 色 |
|--------|---------|-----|
| 総ユーザー数 | 登録されている全ユーザー数 | 青 |
| 有効ユーザー | 現在有効なユーザー数 | 緑 |
| アプリ数 | 登録されている全アプリ数 | 紫 |
| 稼働中アプリ | 現在稼働中のアプリ数 | 青 |
| 未解決インシデント | 対応が完了していないインシデント数 | オレンジ |
| 保留中の変更 | 承認待ちの変更要求数 | 黄色 |

**最近の操作ログ**: 直近10件の操作履歴

**アプリ一覧**: 登録アプリの概要

#### 4.2.4 基本操作（CRUD）

**Create（作成）**:
1. 各画面で「新規〇〇」ボタンをクリック
2. モーダルフォームが表示される
3. 必須項目（赤い*マーク）を入力
4. 「保存」ボタンをクリック
5. 成功メッセージが表示される

**Read（読取）**:
- 一覧画面に自動表示される
- 検索ボックスでフィルタ可能

**Update（更新）**:
1. 一覧の「✏️」（編集）ボタンをクリック
2. 編集フォームが表示される
3. 変更したい項目を修正
4. 「保存」ボタンをクリック

**Delete（削除）**:
1. 一覧の「🗑️」（削除）ボタンをクリック
2. 確認ダイアログが表示される
3. 「OK」をクリックで削除実行
4. **注意**: 削除は取り消せません

#### 4.2.5 検索・フィルタ機能

**検索ボックス**:
- キーワードを入力してEnterキー
- 名前、説明等の部分一致で検索

**フィルタ**:
- ステータス、権限、カテゴリ等で絞り込み
- プルダウンメニューから選択

**ソート**:
- 列のヘッダーをクリックで昇順/降順切替

## 6. セッション4: インシデント管理（30分）

### 6.1 学習目標

- インシデントとは何かを理解する
- インシデントを登録できる
- インシデントのステータスを更新できる
- 優先度を正しく設定できる

### 6.2 講義内容

#### 6.2.1 インシデントとは

**定義**: アプリの障害・不具合・問い合わせ事象

**目的**: 問題を記録し、解決まで追跡する

**ライフサイクル**:
```
[オープン] → [対応中] → [解決済み] → [クローズ]
```

#### 6.2.2 優先度の設定

| 優先度 | 状況 | 対応目標 | 例 |
|--------|------|---------|-----|
| **高** | 業務が停止している | 4時間以内 | システムダウン、ログイン不可 |
| **中** | 業務に支障があるが継続可能 | 24時間以内 | 一部機能が動かない |
| **低** | 軽微な問題・改善要望 | 72時間以内 | 表示が崩れる、誤字修正 |

#### 6.2.3 インシデント登録

**必須項目**:
- タイトル: 問題の概要
- 対象アプリ: 関連するアプリ
- 優先度: 高/中/低
- 説明: 詳細な状況

**任意項目**:
- 報告者: 報告した人の名前
- 担当者: 対応担当者

**良いタイトルの例**:
- ✅ 「勤怠管理アプリでログインできない」
- ✅ 「経費精算の承認ボタンが反応しない」
- ❌ 「エラー」（具体性がない）
- ❌ 「動かない」（何が動かないか不明）

**良い説明の例**:
```
【症状】
勤怠管理アプリにログインしようとすると、
「認証エラー」と表示されてログインできない。

【再現手順】
1. 勤怠管理アプリを開く
2. ユーザー名・パスワードを入力
3. ログインボタンをクリック
4. 「認証エラー」と表示される

【影響範囲】
全社員（約50名）

【発生時刻】
2026年2月4日 9:00頃から
```

#### 6.2.4 ステータス更新

**ステータス遷移**:

```
オープン（新規登録）
  ↓
対応中（担当者が対応開始）
  ↓
解決済み（問題が解決）
  ↓
クローズ（報告者が確認完了）
```

**誰が更新するか**:
- オープン → 対応中: 担当者
- 対応中 → 解決済み: 担当者
- 解決済み → クローズ: 報告者または管理者

### 6.3 実習

**課題1: インシデント登録（練習用）**

以下のシナリオでインシデントを登録してください:

**シナリオ**:

- 対象アプリ: 営業管理
- 優先度: 高（業務に大きな支障）
- 報告者: あなたの名前
- 担当者: （未割当）
- 説明: 上記シナリオの内容を記載

**課題2: インシデント一覧確認**
1. インシデント管理画面で登録したインシデントを確認

## 7. セッション5: 変更管理（15分）

### 7.1 学習目標

- 変更要求とは何かを理解する
- 変更要求を作成できる
- 承認フローを理解する

### 7.2 講義内容

#### 7.2.1 変更要求とは

**定義**: アプリの機能追加・改善・修正の申請

**インシデントとの違い**:
- インシデント: 既に発生した問題の対応
- 変更要求: 新しい機能や改善の申請（計画的）

#### 7.2.2 変更タイプ

| タイプ | 説明 | 例 |
|--------|------|-----|
| **機能追加** | 新しい機能を追加 | 「承認履歴を表示する機能を追加」 |
| **機能変更** | 既存機能の仕様変更 | 「承認ルートを3段階に変更」 |
| **バグ修正** | 不具合の修正 | 「計算式の誤りを修正」 |
| **改善** | パフォーマンス等の改善 | 「検索速度を改善」 |

#### 7.2.3 承認フロー

```
下書き（作成者が編集可能）
  ↓
承認待ち（管理者に通知）
  ↓
承認済み/却下（管理者が判断）
  ↓
実装中（開発者が作業）
  ↓
完了（リリース済み）
```

#### 7.2.4 変更要求作成

**必須項目**:
- タイトル: 変更内容の概要
- 対象アプリ: 関連するアプリ
- タイプ: 機能追加/変更/修正/改善
- 説明: 詳細な変更内容と理由

**任意項目**:
- 予定実施日: 変更を実施したい日

**良い説明の例**:
```
【変更内容】
経費精算アプリに、領収書画像を添付する機能を追加したい。

【理由】
現在は紙の領収書を別途提出する必要があり、
手間がかかっている。画像添付機能があれば、
承認者がその場で確認でき、業務効率が向上する。

【期待効果】
- 承認時間が平均30分短縮
- 紙の保管・管理コストが削減
```

### 7.3 実習

**課題1: 変更要求作成**

以下のシナリオで変更要求を作成してください:

**シナリオ**:
勤怠管理アプリで、月次の勤怠データをExcelファイルで

**登録内容**:
- タイトル: 勤怠データExcelエクスポート機能の追加
- 対象アプリ: 勤怠管理
- タイプ: 機能追加
- 説明: 上記シナリオの内容を記載
- 予定実施日: 来月1日

**課題2: ステータス変更**
1. 作成した変更要求のステータスを「承認待ち」に変更
2. 一覧で「承認待ち」のみをフィルタ表示

## 8. セッション6: 監査ログ・設定（15分）

### 8.1 学習目標

- 監査ログの目的を理解する
- ログを検索・フィルタできる
- ログをエクスポートできる
- バックアップを取得できる

### 8.2 講義内容

#### 8.2.1 監査ログとは

**定義**: システム上で行われた全操作の履歴記録

**記録される情報**:
- 日時
- ユーザー
- 操作タイプ（ログイン、作成、更新、削除等）
- 対象（ユーザー、アプリ、インシデント等）
- 詳細情報

**目的**:
- セキュリティ監視
- 問題発生時の原因調査
- コンプライアンス対応

#### 8.2.2 ログ検索

**検索条件**:
- 日付範囲（From〜To）
- 操作タイプ
- 対象タイプ
- ユーザー

**検索例**:
- 「昨日のログイン操作のみ」
- 「特定ユーザーの削除操作」
- 「先月のインシデント作成操作」

#### 8.2.3 ログエクスポート

**用途**:
- 長期保管
- 外部分析ツールでの分析
- 監査対応

**手順**:
1. 検索条件で対象を絞り込み
2. 「エクスポート」ボタンをクリック
3. CSVファイルがダウンロードされる

**ファイル名**: `audit_log_YYYYMMDD_HHMMSS.csv`

#### 8.2.4 バックアップ・リストア

**バックアップ**:
1. システム設定→バックアップタブ
2. 「バックアップ作成」ボタンをクリック
3. JSONファイルがダウンロードされる
4. 安全な場所に保管

**バックアップの内容**:
- 全ユーザー情報
- 全アプリ情報
- 全インシデント
- 全変更要求
- 全監査ログ
- システム設定

**リストア**:
1. 「バックアップファイル」でファイルを選択
2. 「リストア実行」をクリック
3. 確認ダイアログで「OK」
4. ページが自動リロード

⚠️ **注意**: リストアは現在のデータを全て上書きします

### 8.3 実習

**課題1: ログ検索**
1. 監査ログ画面を開く
2. 今日の操作ログを検索
3. 「作成」操作のみをフィルタ表示

**課題2: ログエクスポート**
1. 今日のログをエクスポート
2. ダウンロードしたCSVファイルを確認

**課題3: バックアップ取得（管理者のみ）**
1. システム設定→バックアップタブを開く
2. バックアップを作成
3. ファイル名とサイズを確認

タイプ: 改善
説明: トレーニング用のテストデータです。

1. 監査ログで自分の操作履歴を確認

### 9.2 応用課題（時間がある方向け）

#### 課題6: ワークフロー体験

#### 課題7: データ分析
1. ダッシュボードで全体の統計を確認
2. 未解決インシデントが多い場合、原因を考察

#### 課題8: エクスポート
1. インシデント一覧をエクスポート（仮想）
2. ログをエクスポート

### 9.3 管理者向け課題

#### 課題9: ユーザー管理
1. テストユーザーを3名登録
2. うち1名を「無効」ステータスに変更
3. 有効ユーザーのみをフィルタ表示

#### 課題10: バックアップ・リストア
1. 現在の状態でバックアップを作成
2. テストデータをいくつか削除
3. バックアップからリストア
4. データが復元されたことを確認
