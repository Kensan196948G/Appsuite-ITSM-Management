## UAT-003: インシデント起票から解決まで ⏸

**所要時間**: 15分
**優先度**: 高

### 前提条件

- [ ] 管理者としてログイン済み
- [ ] ダッシュボードが表示されている

### テスト手順

#### ステップ1: インシデント一覧表示

- [ ] サイドバーの「インシデント管理」をクリック
- [ ] インシデント一覧が表示される
- [ ] 既存インシデント件数を確認: ______件

**期待結果**: インシデント一覧テーブルが表示される

#### ステップ2: 新規インシデント作成

- [ ] 「新規インシデント」ボタンをクリック
- [ ] モーダルが表示される
- [ ] 以下の情報を入力:
  - [ ] タイトル: `UATテストインシデント`
  - [ ] 対象アプリ: `勤怠管理` を選択
  - [ ] 優先度: `高` を選択
  - [ ] 説明: `テスト用のインシデント` を入力
- [ ] 「保存」ボタンをクリック
- [ ] モーダルが閉じる
- [ ] トースト通知「インシデントを作成しました」が表示される
- [ ] 一覧に新しいインシデントが追加される
- [ ] 新規インシデントのステータスが「オープン」である

**期待結果**: インシデントが正常に作成され、一覧に表示される

#### ステップ3: ステータスを「対応中」に変更

- [ ] 作成したインシデントの「編集」ボタン（鉛筆アイコン）をクリック
- [ ] 編集モーダルが表示される
- [ ] ステータスを「対応中」に変更
- [ ] 担当者を選択（任意のユーザー）
- [ ] 「保存」ボタンをクリック
- [ ] トースト通知が表示される
- [ ] 一覧のステータスが「対応中」に更新される

**期待結果**: ステータスが「オープン」→「対応中」に遷移

#### ステップ4: ステータスを「解決済」に変更

- [ ] 再度「編集」ボタンをクリック
- [ ] ステータスを「解決済」に変更
- [ ] 「保存」ボタンをクリック
- [ ] トースト通知が表示される
- [ ] 一覧のステータスが「解決済」に更新される

**期待結果**: ステータスが「対応中」→「解決済」に遷移

#### ステップ5: ステータスを「クローズ」に変更

- [ ] 再度「編集」ボタンをクリック
- [ ] ステータスを「クローズ」に変更
- [ ] 「保存」ボタンをクリック
- [ ] トースト通知が表示される
- [ ] 一覧のステータスが「クローズ」に更新される

**期待結果**: ステータスが「解決済」→「クローズ」に遷移

#### ステップ6: ワークフロー検証

- [ ] ワークフローが以下の順序で遷移したことを確認:
  ```
  オープン → 対応中 → 解決済 → クローズ
  ```

#### ステップ7: 通知確認

- [ ] 高優先度インシデント作成時に通知が発生したか確認
- [ ] ヘッダーの通知ベル（🔔）に未読バッジがあるか確認
- [ ] 通知ベルをクリックして通知パネルを開く
- [ ] インシデント作成の通知が表示されているか確認

**期待結果**: 高優先度インシデントの通知が表示される

#### ステップ8: 監査ログ確認

- [ ] サイドバーの「監査ログ」をクリック
- [ ] 以下の操作が記録されているか確認:
  - [ ] インシデント作成（create）
  - [ ] ステータス更新（update）× 3回
- [ ] 操作タイプフィルタで「作成」を選択
- [ ] インシデント作成ログのみ表示される

**期待結果**: すべての操作が監査ログに記録されている

### 合格基準

- [ ] インシデント作成成功
- [ ] ステータス遷移が正常（ワークフロールール遵守）
- [ ] 担当者割り当て動作
- [ ] 通知が発生（高優先度の場合）
- [ ] 監査ログに全操作記録

### 判定

- [ ] ✅ 合格
- [ ] ❌ 不合格（理由: ____________________）

## UAT-004: 変更要求の承認フロー ⏸

**所要時間**: 10分
**優先度**: 高

### 前提条件

- [ ] 管理者としてログイン済み
- [ ] ダッシュボードが表示されている

### テスト手順

#### ステップ1: 変更管理セクションへ移動

- [ ] サイドバーの「変更管理」をクリック
- [ ] 変更要求一覧が表示される
- [ ] 既存変更要求件数を確認: ______件

**期待結果**: 変更要求一覧が表示される

#### ステップ2: 新規変更要求作成

- [ ] 「新規変更要求」ボタンをクリック
- [ ] モーダルが表示される
- [ ] 以下の情報を入力:
  - [ ] タイトル: `UAT機能追加要求`
  - [ ] 対象アプリ: `顧客管理` を選択
  - [ ] 種別: `機能追加` を選択
  - [ ] 実施予定日: 来週の日付を選択
  - [ ] 説明: `UAT用のテスト変更要求`
- [ ] 「保存」ボタンをクリック
- [ ] トースト通知が表示される
- [ ] 一覧に新しい変更要求が追加される
- [ ] ステータスが「下書き」である

**期待結果**: 変更要求が「下書き」ステータスで作成される

#### ステップ3: 下書き → 申請中

- [ ] 作成した変更要求の「編集」ボタンをクリック
- [ ] ステータスを「申請中」に変更
- [ ] 「保存」ボタンをクリック
- [ ] トースト通知が表示される
- [ ] ステータスが「申請中」に更新される

**期待結果**: ステータスが「下書き」→「申請中」に遷移

#### ステップ4: 申請中 → 承認

- [ ] 再度「編集」ボタンをクリック
- [ ] ステータスを「承認」に変更
- [ ] 「保存」ボタンをクリック
- [ ] ステータスが「承認」に更新される

**期待結果**: ステータスが「申請中」→「承認」に遷移

#### ステップ5: 承認 → 実施中

- [ ] 再度「編集」ボタンをクリック
- [ ] ステータスを「実施中」に変更
- [ ] 「保存」ボタンをクリック
- [ ] ステータスが「実施中」に更新される

**期待結果**: ステータスが「承認」→「実施中」に遷移

#### ステップ6: 実施中 → 完了

- [ ] 再度「編集」ボタンをクリック
- [ ] ステータスを「完了」に変更
- [ ] 「保存」ボタンをクリック
- [ ] ステータスが「完了」に更新される

**期待結果**: ステータスが「実施中」→「完了」に遷移

#### ステップ7: ワークフロー検証

- [ ] ワークフローが以下の順序で遷移したことを確認:
  ```
  下書き → 申請中 → 承認 → 実施中 → 完了
  ```

#### ステップ8: 不正遷移のテスト

- [ ] 「新規変更要求」で新しい変更要求を作成（ステータス: 下書き）
- [ ] 編集モーダルを開く
- [ ] ステータスを「完了」に直接変更しようとする
- [ ] 保存時にエラーメッセージが表示されるか確認
  - または、不正な遷移がブロックされることを確認

**期待結果**: 下書き→完了の直接遷移がブロックされる

#### ステップ9: 監査ログ確認

- [ ] 「監査ログ」セクションに移動
- [ ] 変更要求の作成・更新操作が記録されているか確認
- [ ] 対象フィルタで「変更要求」を選択
- [ ] 変更要求関連の操作のみ表示される

**期待結果**: すべての操作が監査ログに記録されている

### 合格基準

- [ ] 変更要求作成成功
- [ ] 承認フロー遷移が正常
- [ ] 不正な遷移（例: 下書き→完了）がブロックされる
- [ ] 監査ログに全操作記録

### 判定

- [ ] ✅ 合格
- [ ] ❌ 不合格（理由: ____________________）

## UAT-010: レスポンシブ表示確認 ⏸

**所要時間**: 5分
**優先度**: 低

### 前提条件

- [ ] 管理者としてログイン済み
- [ ] ダッシュボードが表示されている
- [ ] ブラウザのDevToolsを使用可能

### テスト手順

#### ステップ1: DevTools準備

- [ ] F12キーでDevToolsを開く
- [ ] Ctrl+Shift+M（またはCmd+Shift+M）でデバイスツールバーを有効化
- [ ] レスポンシブモードになることを確認

**期待結果**: デバイスツールバーが表示される

#### ステップ2: デスクトップ表示確認（1920×1080）

- [ ] デバイスツールバーで「Responsive」を選択
- [ ] 画面サイズを 1920 × 1080 に設定
- [ ] 以下を確認:
  - [ ] サイドバーが左側に常時表示されている
  - [ ] ダッシュボードの統計カードが横に並んでいる
  - [ ] チャートが適切に表示されている
  - [ ] テーブルが画面幅に収まっている
  - [ ] ヘッダーが上部に固定されている

**期待結果**: デスクトップレイアウトで全要素が適切に表示

#### ステップ3: タブレット表示確認（768×1024）

- [ ] デバイスツールバーで「iPad」を選択（または768×1024に設定）
- [ ] 以下を確認:
  - [ ] サイドバーが初期状態で非表示または折りたたまれている
  - [ ] ハンバーガーメニューボタン（☰）が表示されている
  - [ ] ハンバーガーメニューをクリックしてサイドバーが表示される
  - [ ] 統計カードが適切に折り返されている
  - [ ] テーブルに横スクロールバーが表示される（必要に応じて）
  - [ ] モーダルが画面幅に収まっている

**期待結果**: タブレットレイアウトで適切に調整される

#### ステップ4: モバイル表示確認（375×667）

- [ ] デバイスツールバーで「iPhone 6/7/8」を選択（または375×667に設定）
- [ ] 以下を確認:
  - [ ] サイドバーが完全に非表示
  - [ ] ハンバーガーメニューボタンが表示
  - [ ] ハンバーガーメニューをクリックしてサイドバーがオーバーレイ表示
  - [ ] 統計カードが縦に並んでいる
  - [ ] テーブルが横スクロール可能
  - [ ] フォーム入力フィールドが画面幅に収まっている
  - [ ] ボタンが指で押しやすいサイズ（最低44px×44px）
  - [ ] モーダルが画面幅に収まり、スクロール可能

**期待結果**: モバイルレイアウトで適切に調整される

#### ステップ5: 各セクションでの表示確認

デスクトップ表示（1920×1080）に戻してから、以下の各セクションをタブレット・モバイルサイズで確認:

##### ユーザー管理

- [ ] タブレット表示:
  - [ ] テーブルが横スクロール可能
  - [ ] 検索ボックスとフィルタが表示される
- [ ] モバイル表示:
  - [ ] ツールバーのボタンが縦に並ぶまたは適切に折り返される
  - [ ] テーブルが横スクロール可能

##### インシデント管理

- [ ] タブレット表示:
  - [ ] フィルタが適切に配置される
  - [ ] テーブルが横スクロール可能
- [ ] モバイル表示:
  - [ ] ツールバーが折り返される
  - [ ] テーブルカラムが省略されず、横スクロールで全て確認可能

##### システム設定

- [ ] タブレット表示:
  - [ ] タブが適切に表示される
  - [ ] フォームが読みやすい
- [ ] モバイル表示:
  - [ ] タブが縦スタックまたは横スクロール
  - [ ] フォーム要素が縦に並ぶ

**期待結果**: 全セクションで各画面サイズに適したレイアウト

#### ステップ6: タッチ操作確認（オプション）

タッチデバイス（タブレット・スマートフォン）がある場合:

- [ ] 実機でシステムを開く
- [ ] タッチでナビゲーション操作
- [ ] ピンチズームが無効化されているか確認
- [ ] スワイプ操作が快適か確認
- [ ] ボタンがタップしやすいサイズか確認

**期待結果**: タッチ操作が快適に動作

### 合格基準

- [ ] デスクトップ表示正常
- [ ] タブレット表示正常
- [ ] モバイル表示正常
- [ ] タッチ操作対応（実機テストした場合）

### 判定

- [ ] ✅ 合格
- [ ] ❌ 不合格（理由: ____________________）
