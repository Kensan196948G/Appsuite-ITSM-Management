## 1. 概要

### 1.1 Appsuiteとは

AppsuiteはITサービスマネジメント（ITSM）を効率的に運用するための統合プラットフォームです。ITIL®フレームワークに準拠し、IT部門がサービス品質を維持・向上させるための機能を提供します。

### 1.2 ITSMの目的

ITSMは、ITサービスの計画、提供、運用、改善を体系的に管理するためのアプローチです。主な目的は以下の通りです：

- ビジネスニーズに合致したITサービスの提供
- サービス品質の継続的な改善
- コスト効率の最適化
- リスクの軽減とコンプライアンスの確保

### 1.3 本ドキュメントの範囲

本ドキュメントでは、Appsuiteで管理する主要なITSMプロセスについて、IT管理者向けに概要と運用フローを説明します。

## 2. インシデント管理

### 2.1 目的

インシデント管理の目的は、ITサービスの中断や品質低下が発生した際に、可能な限り迅速にサービスを復旧させることです。

### 2.2 インシデントの定義

インシデントとは、ITサービスの計画外の中断、またはITサービス品質の低下を指します。

### 2.3 プロセスフロー

```
[検知・報告] → [登録・分類] → [優先度設定] → [調査・診断] → [解決・復旧] → [クローズ]
```

**ステップ詳細:**

| ステップ | 担当者 | 主な作業内容 |
|----------|--------|--------------|
| 検知・報告 | ユーザー/監視システム | インシデントの発見と報告 |
| 登録・分類 | サービスデスク | チケット作成、カテゴリ・影響範囲の設定 |
| 優先度設定 | サービスデスク | 影響度×緊急度による優先度決定 |
| 調査・診断 | 技術チーム | 原因特定と解決策の検討 |
| 解決・復旧 | 技術チーム | 対処の実施とサービス復旧確認 |
| クローズ | サービスデスク | ユーザー確認後、チケットのクローズ |

### 2.4 優先度マトリクス

| | 緊急度：高 | 緊急度：中 | 緊急度：低 |
|---|---|---|---|
| **影響度：高** | P1（最優先） | P2（高） | P3（中） |
| **影響度：中** | P2（高） | P3（中） | P4（低） |
| **影響度：低** | P3（中） | P4（低） | P5（計画対応） |

### 2.5 エスカレーションルール

- **P1インシデント:** 15分以内に担当者アサイン、1時間以内に経営層へ報告
- **P2インシデント:** 30分以内に担当者アサイン、4時間以内に進捗報告
- **P3インシデント:** 2時間以内に担当者アサイン

## 3. 問題管理

### 3.1 目的

問題管理の目的は、インシデントの根本原因を特定・解決し、将来のインシデント発生を防止することです。

### 3.2 問題の定義

問題とは、1つ以上のインシデントの原因となる未知の根本原因を指します。

### 3.3 プロセスフロー

```
[問題の特定] → [問題の登録] → [根本原因分析] → [解決策の立案] → [恒久対策の実施] → [クローズ]
```

**ステップ詳細:**

| ステップ | 担当者 | 主な作業内容 |
|----------|--------|--------------|
| 問題の特定 | 問題管理者 | 繰り返しインシデントの分析、トレンド把握 |
| 問題の登録 | 問題管理者 | 問題チケットの作成、関連インシデントの紐付け |
| 根本原因分析 | 技術チーム | RCA（Root Cause Analysis）の実施 |
| 解決策の立案 | 技術チーム | ワークアラウンドまたは恒久対策の検討 |
| 恒久対策の実施 | 技術チーム | 変更管理プロセスを通じた対策の実装 |
| クローズ | 問題管理者 | 効果確認後、問題チケットのクローズ |

### 3.4 既知エラーデータベース（KEDB）

特定された問題とそのワークアラウンドは、既知エラーデータベース（KEDB）に登録し、サービスデスクが迅速に対応できるよう情報共有します。

## 4. 変更管理

### 4.1 目的

変更管理の目的は、ITサービスへの変更を標準化された方法で管理し、変更に伴うリスクを最小限に抑えることです。

### 4.2 変更の種類

| 変更タイプ | 説明 | 承認プロセス |
|------------|------|--------------|
| 標準変更 | リスクが低く、事前承認された定型変更 | 事前承認済み |
| 通常変更 | 一般的な変更要求 | CAB承認 |
| 緊急変更 | 重大インシデント対応のための緊急変更 | 緊急CAB承認 |

### 4.3 プロセスフロー

```
[変更要求(RFC)] → [評価・分類] → [CAB審議] → [承認/却下] → [実施計画] → [実装] → [レビュー] → [クローズ]
```

**ステップ詳細:**

| ステップ | 担当者 | 主な作業内容 |
|----------|--------|--------------|
| 変更要求(RFC) | 申請者 | 変更内容、理由、影響範囲の記載 |
| 評価・分類 | 変更管理者 | リスク評価、変更タイプの決定 |
| CAB審議 | 変更諮問委員会 | 技術的・ビジネス的観点からの審議 |
| 承認/却下 | CAB/変更管理者 | 変更の承認または却下判定 |
| 実施計画 | 実装チーム | 詳細な実施手順、ロールバック計画の策定 |
| 実装 | 実装チーム | 計画に基づく変更の実施 |
| レビュー | 変更管理者 | 実施結果の確認、PIR（実装後レビュー）の実施 |
| クローズ | 変更管理者 | 変更チケットのクローズ |

### 4.4 変更諮問委員会（CAB）

CABは変更要求を評価・承認する委員会です。通常、以下のメンバーで構成されます：

- 変更管理者（議長）
- 技術リーダー
- サービスオーナー
- セキュリティ担当者
- 必要に応じてステークホルダー

## 5. サービス要求管理

### 5.1 目的

サービス要求管理の目的は、ユーザーからの標準的なサービス要求を効率的に処理することです。

### 5.2 サービス要求の例

- パスワードリセット
- ソフトウェアインストール申請
- アクセス権限の付与・変更
- 機器の貸出申請
- 情報提供の依頼

### 5.3 プロセスフロー

```
[要求の受付] → [要求の確認] → [承認（必要時）] → [フルフィルメント] → [クローズ]
```

**ステップ詳細:**

| ステップ | 担当者 | 主な作業内容 |
|----------|--------|--------------|
| 要求の受付 | サービスデスク | セルフサービスポータルまたは電話での受付 |
| 要求の確認 | サービスデスク | 要求内容の妥当性確認 |
| 承認 | 承認者 | 必要に応じて上長承認を取得 |
| フルフィルメント | 担当チーム | 要求の実行（作業実施） |
| クローズ | サービスデスク | ユーザー確認後、チケットのクローズ |

### 5.4 サービスカタログ

主要なサービス要求はサービスカタログに定義し、標準的な処理手順とSLAを設定します。

## 6. 構成管理（CMDB）

### 6.1 目的

構成管理の目的は、ITサービスを構成する資産（構成アイテム）の情報を正確に把握・管理することです。

### 6.2 構成アイテム（CI）の例

- ハードウェア（サーバー、ネットワーク機器、PC等）
- ソフトウェア（OS、アプリケーション、ライセンス）
- ドキュメント（設計書、手順書）
- サービス（業務アプリケーション）

### 6.3 CMDBの主要属性

| 属性 | 説明 |
|------|------|
| CI ID | 一意の識別子 |
| CI名 | 構成アイテムの名称 |
| タイプ | ハードウェア/ソフトウェア等の分類 |
| ステータス | 稼働中/保守中/廃棄等 |
| 所有者 | 責任者 |
| 関連CI | 依存関係にあるCI |
| ロケーション | 設置場所 |

### 6.4 構成管理プロセス

```
[識別] → [制御] → [ステータス管理] → [検証・監査]
```

- **識別:** CIの定義とCMDBへの登録
- **制御:** CIの変更を変更管理プロセスと連携して管理
- **ステータス管理:** CIのライフサイクルを通じた状態管理
- **検証・監査:** 定期的なCMDB情報の正確性確認

## 7. SLA管理

### 7.1 目的

SLA管理の目的は、ITサービスの品質目標を定義し、合意されたサービスレベルを維持・監視することです。

### 7.2 SLAの構成要素

| 要素 | 説明 |
|------|------|
| サービス内容 | 提供するサービスの範囲と説明 |
| サービス時間 | サービス提供時間（例：24時間365日） |
| 可用性目標 | システム稼働率（例：99.9%） |
| 応答時間 | インシデント対応開始までの時間 |
| 解決時間 | インシデント解決までの時間 |
| エスカレーション | 問題発生時の連絡経路 |

### 7.3 SLA目標例（インシデント対応）

| 優先度 | 応答時間 | 解決目標時間 |
|--------|----------|--------------|
| P1 | 15分 | 4時間 |
| P2 | 30分 | 8時間 |
| P3 | 2時間 | 24時間 |
| P4 | 4時間 | 72時間 |
| P5 | 8時間 | 1週間 |

### 7.4 SLAモニタリング

AppsuiteのダッシュボードでSLA達成率をリアルタイムに監視できます。定期的なSLAレポートを作成し、以下の指標を追跡します：

- インシデント解決率（SLA内/SLA外）
- 平均応答時間（MTTR: Mean Time To Respond）
- 平均解決時間（MTRS: Mean Time To Resolve）
- 顧客満足度スコア

### 7.5 継続的改善

SLA未達成が発生した場合は、以下のステップで改善を図ります：

1. 未達成原因の分析
2. 改善策の立案
3. アクションプランの実行
4. 効果の測定・評価
